The Inheritance — Game Loop Architecture

Aur / Naer-Reth / The Inheritance  ·  Engine: Godot 4.5  ·  Format: 3D Single-Player  ·  Last Updated: 2026-02-21

What the player does

Spends: Time, fuel, suit battery, ship power budget, risk exposure
Gets: Resources, ship upgrades, automation tools, new biome access, Mega-Project progress
Loss condition: Ship Integrity = 0 (tech tree + schematics persist on respawn)
Win condition: Naer-Reth activation (Cycle 1) / Regulator interface (Cycle 2)
No: Starvation timers, forced busywork, dead-end tech nodes

Three Design Pillars that Shape the Loop

  • Scanner-First: Discover before acting. Blind harvesting is inefficient by design.
  • Automation as Reward: Earned through tech tree — never given. Manual play always outperforms drones on a per-deposit basis.
  • Nomadic Survival: The ship moves constantly. Settling in place is never the goal. Every move is purposeful.

Ship Global Variables — The Living Ship

Module System — Weight/Performance Formula

Total Ship Weight = Hull Base Weight + Σ(all module weights) Fuel Consumption = Base Rate × (1 + Weight Modifier) Top Speed = Base Speed × (1 − Weight Modifier) Weight Modifier = (Total Weight − Optimal Weight) / Optimal Weight [clamped to 0 at or below Optimal Weight]

Every module added = trade-off. There is no optimal loadout — only choices.

Player Suit — Separate Upgrade Track

The suit is the player's personal equipment. Its upgrade track is independent of the ship tech tree — suit upgrades do not compete with ship resources.

AxisEffect
Battery CapacityMax field time before recharge; determines how long tools can fire
Movement SpeedBase movement; also reduces severity of 0%-battery speed penalty
Scanner Range/QualityPlayer's baseline scanner — works within the Scanner Array module's detection ceiling
Armor / ResistanceReduces damage from hazards and enemy attacks

Scanner Interaction: Ship Scanner Array sets the hard ceiling (what tiers can be detected). Suit scanner upgrade improves baseline range and resolution within that ceiling. Both axes matter independently.

Battery & Field Loop: Suit charges at ship (fast). Spare batteries occupy general inventory slots — carrying more spares = less cargo space. This is the core field-trip trade-off. Tools don't fire at 0%; movement speed −25% (never immobilized).

World generation: Biome type sets the resource palette. Biome tier sets purity & depth distribution. No two worlds identical — meta-rules guarantee all Mega-Project materials appear somewhere.

Resource Palette Structure

  • Primary resource: Most common; low-purity sources abundant
  • Secondary resource: Moderate presence; mixed purity
  • Rare resource: Uncommon; deep deposits or specific features
  • Surface collectibles: Visible without scanner; walk-up pickup

Deposit Chains: High-tier scanners can detect adjacent lateral chains extending through the sub-surface. Exploiting a full chain may require repositioning the ship or deploying drones. Incentivizes staying in one zone vs. always moving on.

First Cycle — "The Tragedy" (Unavoidable)

  • The Naer-Reth does revive the planet's ecology — it succeeds
  • It also drains all energy from Aur's surface as fuel (geological, technological, biological)
  • The Serev were consumed during its first partial activation — that is what destroyed them
  • The player, the ship, and Rel are consumed during the full activation
  • The player cannot avoid this ending on the First Cycle — only understand it before or after

Second Cycle — "The True Ending" (Earned)

  • Accessible only after completing the First Cycle (save flag gate — no skill check)
  • The Regulator limits the Naer-Reth's drain to geological/solar sources only — excluding biological
  • The Serev knew the Regulator could exist. They left instructions. The player was the intended recipient.
  • Harder than the tragedy: 2× total resources, parallel construction tracks, all biome tiers revisited
  • Aur lives. The player lives. Rel understands, finally and completely.

InputManager Architecture — Centralized Routing

Player Input (Keyboard/Mouse OR Gamepad) ↓ InputManager (autoload — persistent) ├─→ PlayerFirstPerson controller (if Context 1 active) ├─→ PlayerThirdPerson controller (if Context 2 active) └─→ UI System (always active when menu open) Rules: • Only ONE controller active at any time (first-person XOR third-person) • Menu input is ALWAYS processed regardless of controller state • Controllers call InputManager.is_action_pressed() — NEVER Input.is_action_pressed() directly • Input remapping updates Godot project settings dynamically via InputManager • Dead zones handled in InputManager, not in individual controllers

View Mode Switching (Tab / Menu)

  • Current view deactivates (freeze character / ship)
  • Camera transitions smoothly over 0.5 seconds
  • Control scheme switches immediately after transition
  • Velocity preserved on switch (walking momentum / ship velocity)
  • Mid-action: action completes in current context; switch queued after
  • Rapid spam prevention: 0.3s debounce cooldown on Tab/Menu

Seamless Device Switching

  • Gamepad → KB/Mouse: any key press or mouse movement
  • KB/Mouse → Gamepad: any button or stick movement beyond dead zone
  • Switch is immediate — no menu, no confirmation, no game pause
  • UI button hints update instantly (E vs X / ● etc.)
  • Activity debounced: 0.1s window to prevent false switching
  • Mouse movement alone (no click) counts as activity

Gamepad First-Class Requirements

  • All actions rebindable including gamepad layouts
  • Analog triggers are pressure-sensitive [0.0–1.0], not binary
  • Left Stick = movement/camera primary (context-dependent)
  • Right Stick = camera/rotation secondary (context-dependent)
  • Sensitivity configurable per axis (X and Y separately)
  • All UI legible at TV viewing distance
  • Radial scanner wheel designed for analog stick input first
